# Why modules?
Javascript used to be relatively simple. ``` <a onclick="alert('gimme pizza')">helll yahhh</a> ``` Note: - Our JavaScript applications used to be relatively simple - Existed on "one line" - Handled primarily simple interactivity, form validation, etc.
But our applications are much more ambitious now. ``` {{#link-to 'pizza'}}helll yahhh{{/link-to}} ``` Note: - But our applications are becoming more ambitous - This demands a whole new level of structure and organization to our code
Note: Most of my code samples deal with pizza for some reason so‚Ä¶get ready
`import` and `export` have been reserved keywords in Javascript from the beginning Note: The words 'import' and 'export' were defined as reserved keywords in the original javascript specification (It would be interesting to know why that was?)
But Javascript had no built in system for modularity Note: but JS has never had a built in system for handling modularity If a developer working on a more complex application wanted to organize their code in a more modular fashion which means that code was shared across the global scope
‚Ä¶with no interface or namespacing, creating headaches and barriers to collaboration ```javascript // utils.js function slice(arr, n) { // ... } ``` ```html <!-- index.html --> <script src="utils.js"></script> <script src="pizza.js"></script> <script> var slice = new Slice('plain'); // conflicts with slice in utils.js </script> ``` Note: This limits stablity and maintainablity of your code, and makes it difficult to collaborate with others, etc.
Now we have multiple ways of handling modules. ``` // commonjs class Pizza { // ... } module.exports.Pizza = Pizza; // amd define(['Pizza'], function () { class Pizza { // ... } return Pizza; }); ``` Note: Eventually, javascript developers solved this problem on their own, by creating their own standards and libraries for handling modules Unfortunately, many of these implementations are incompatible with each other, leading to long debates over which modular system is the best, etc., and leading to a complicated package ecosystem
Many languages were able to avoid such a conflict by having a built-in interface for modules ```python # pizza.py class Pizza: def __init__(self, toppings): self.toppings = toppings def Slice: def __init__(self, type): self.type = type # main.py from pizza import Slice ``` Note: Many languages avoided this kind of debate and competition through having a built-in specification and interface
Having a built in specification for Javascript provides one standard for everyone to follow Note: Javascript has one now as of July, with the publication of the modual specification in ES2015. Developers of packages have one way of distributing their code, and users have one way of consuming them But as I'll touch on later, this only creates the potential for cleaning up our package ecology, getting there will likely be more complicated
# ES2015 Module Basics - No `modules` or `define` keywords - Just `import` and `export` - No need to `use strict` - Multiple exports(!) Note: So what are ES2015 modules? The basics - First there's only import and export, no modules or define - Every ES2015 module is in strict-mode by default - And one of the biggest deals about Es2015 is that modules can have more than one export
Exports have to be declared. You can export `function`, `class`, `var`, `const`, and `let`. ```javascript // toppings.js export const combo = ['pineapple', 'feta'], // üòã // pizza.js export class Pizza { constructor(toppings) { this.toppings = toppings; } } ``` ```javascript // index.js import { combo } from './libs/toppings'; import { Pizza } from './libs/pizza'; let pizza = new Pizza(combo); ``` Note: Just like in CommonJS, exports have to be explicity declared. Nothing is exported unless you say it is. Exports have to be named. Exports can be given a type.
You can have multiple `export`s‚Ä¶ ```javascript // toppings.js export const plain = [], export const pepperoni = ['pepperoni'], export const combo1 = ['pineapple', 'feta'], export const combo2 = ['pepperoni', 'sausage', 'people']; ``` Note: You can have all the exports you want in a module
‚Ä¶or you can `export` a list‚Ä¶ ```javascript // toppings.js const plain = [], pepperoni = ['pepperoni'], combo1 = ['pineapple', 'feta'], combo2 = ['pepperoni', 'sausage', 'people']; export { plain, pepperoni, combo1, combo2 } ``` Note: Although, you might think exporting a list of values is more clear
‚Ä¶and rename your `export`s. ```javascript const plain = [], pepperoni = ['pepperoni'], combo1 = ['pineapple', 'feta'], combo2 = ['pepperoni', 'sausage', 'people']; export { plain, pepperoni, combo1 as pineappleAndFeta, combo2 as meatLovers } ``` Note: And for the sake of clarity, you might choose to rename your exports
And you can do the same with `import`s. ```javascript // index.js import { plain as noToppings, meatLovers as meeeat } from './libs/toppings'; import { Pizza } from './libs/pizza'; let pizza1 = new Pizza(noToppings), pizza2 = new Pizza(meeeat); ``` Note: But you might also choose to rename them as their consumed by an import
You can also namespace `import`s as an object ```javascript // index.js import * as toppings from './libs/toppings'; let plain = new Pizza(toppings.plain); ``` Note: You could also do that through namespacing your imports as an object
And you can re-export modules from other dependencies ```javascript export { default as plain } from "./pizzas/plain"; export { default as pepperoni } from "./pizzas/pepperoni"; export { default as pineappleAndFeta } from "./pizzas/pineappleAndFeta"; export { default as meatLovers } from "./pizzas/meatLovers"; ``` Note: There are also times where all you want to do is rename and re-export something from an external library or dependency, in which case you can re-export from another source like so
But what about `default`? Note: There is also a `default` keyword, which has the benefit of providing some backward compatability with older module systems
CommonJS and AMD modules only export once. Note: As i've said, CommonJS and AMD can only export once.
Those exports become equivalent to `default` in ES2015 ```javascript import _ from 'lodash'; // import { default as _ } from 'lodash'; // var _ = require('lodash'); let log = console.log.bind(console); export default log; // export { log as default } // module.exports = console.log.bind(console); ``` Note: So, ES2015 treats those single exports from older modules essentially as the default export, named "default"
# Rules for implementation No nested `import`s or `export`s ```javascript const date = new Date(); if (date.getDay() === 3) { // wednesday is pizza day import { plain } from './all'; // this should fail } ``` Note: For implementing ES2015 modules, there are a few rules First, import and export can be at the top level only
An implementation must load all imported modules recursively. Note: Every import is only completed after all of that modules external dependencies have been loaded This has been the main thing holding up standardization and implementation of the module specification
Linking ``` // all.js export { default as plain } from "./pizzas/plain"; export { default as pepperoni } from "./pizzas/pepperoni"; export { default as pineappleAndFeta } from "./pizzas/pineappleAndFeta"; export { default as meatLovers } from "./pizzas/meatLovers"; // index.js import { taco } from './libs/all.js'; // this should fail ``` ![tacocat](cat.gif) Note: An import can't link to something that isn't being exported

# "Tree-shaking" Rollup only bundles what you need ```javascript // pizza.js export class Pizza { ... } export class Slice { ... } // index.js import { Pizza } from './pizza'; console.log(Pizza(['veggies', 'more veggies'])); // bundle.js export class Pizza { ... } console.log(Pizza(['veggies', 'more veggies'])); ``` Note: One of the especially cool features of rollup is "tree shaking" which allows rollup to produce minimal bundle sizes by only importing the code that's actually *used* [[ demo if there's time!! ]]
